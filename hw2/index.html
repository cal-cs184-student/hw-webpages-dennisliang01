Overview

In this assignment I built a progressive pipeline for geometric modeling, starting from Bézier curve and surface evaluation using de Casteljau's algorithm, then moving into halfedge mesh operations including area-weighted vertex normals, edge flips, and edge splits, and finally combining everything into Loop subdivision. What I found most interesting was how deeply each piece depended on the correctness of the previous one, subdivision only works if splits and flips are right, which only works if every halfedge pointer is meticulously maintained. The assignment gave me a genuine appreciation for the low-level geometric reasoning that underlies the smooth surfaces we take for granted in games and film.

Section I: Bezier Curves and Surfaces

Part 1: Bezier curves with 1D de Casteljau subdivision

De Casteljau's algorithm evaluates a Bézier curve at a parameter t ∈ [0, 1] through repeated linear interpolation over a set of control points. At each step, every adjacent pair of points is blended together using the formula (1 − t) · pi + t · pi+1, producing a new, smaller set of points. This process repeats on each successive result until only a single point remains, that point is the location on the curve at parameter t. In my implementation, evaluateStep performs one such reduction: it iterates over adjacent pairs in the input vector and pushes each interpolated result into a new vector, which is returned. To fully evaluate the curve, this function is called recursively on its own output until the vector contains just one point, which for an n-th degree Bézier curve requires exactly n steps.

Part 2: Bezier surfaces with separable 1D de Casteljau

A Bézier patch extends de Casteljau's algorithm to two dimensions by applying it separably across two parameters, u and v. The control points form an n × n grid, and evaluation proceeds in two passes. In my implementation, evaluateStep performs a single linear interpolation pass identically to the 1D case, while evaluate1D calls it recursively until one point remains, fully evaluating a 1D Bézier curve at parameter t. In evaluate(u, v), each row of the control point grid is first evaluated at parameter u using evaluate1D, producing a single intermediate point per row. These intermediate points are then collected into a new vector and evaluated once more at parameter v, effectively sweeping the curve across the second axis. The result is the point on the surface at (u, v).

Section II: Triangle Meshes and Half-Edge Data Structure

Part 3: Area-weighted vertex normals

To compute an area-weighted vertex normal, I traverse all non-boundary faces adjacent to the vertex using the halfedge data structure, cycling around by repeatedly following twin()->next() until returning to the starting halfedge. For each neighboring face, I retrieve the three vertex positions of that triangle and compute the cross product of two of its edge vectors — (p1 − p0) × (p2 − p0). This cross product naturally encodes both the face normal direction and its area as a magnitude, so larger triangles contribute more to the result. Each such vector is accumulated into a running sum, and once all neighbors have been visited the sum is normalized to produce a unit normal, giving a smooth, area-weighted approximation of the surface normal at that vertex.

Part 4: Edge flip

The edge flip works by treating the two triangles sharing an edge as a diamond of four vertices: vA and vB (the original edge endpoints) and vC and vD (the two apexes). Flipping re-routes the shared edge so it runs between vC and vD instead. No new elements are allocated, every existing halfedge, edge, and face is simply rewired via setNeighbors.

The implementation first collects all ten relevant halfedges (six inner, four outer twins), the four vertices, four outer edges, and two faces before touching anything. Writing all assignments after the collection phase means no pointer is chased after it has already been redirected, which avoids the most common source of corruption in halfedge surgery.

The main debugging trick was drawing the before/after diagrams by hand and labeling each halfedge with its new next/twin/vertex triple before writing a single line of code. Cross-checking those labels against the setNeighbors calls made it easy to spot any mismatch. A secondary check was verifying that every outer halfedge's next and face are left unchanged, only their twin pointers need updating, which kept the outer rewiring minimal and less error-prone.

Part 5: Edge split

Edge split inserts a new midpoint vertex vM at the average of the two edge endpoints, then subdivides the surrounding diamond into four triangles. Unlike a flip, this requires allocating new mesh elements: six new halfedges, three new edges, two new faces, and the midpoint vertex itself. The original two faces are reused for two of the four resulting triangles to minimize allocation overhead.

The boundary case is handled separately since only one triangle exists, the split produces two interior triangles plus a new boundary loop halfedge wired to a freshly allocated boundary face. The interior case follows the same collect-then-reassign discipline as the flip: all existing pointers are gathered into named locals first, new elements are allocated second, and every setNeighbors call is written only after the full picture is known.

The most useful debugging trick was distinguishing which new edges should be flagged isNew = true versus false. Only the two "spoke" edges that cross the original edge boundary are truly new for the purposes of Loop subdivision's flip step, the third new edge that runs parallel to an original edge is marked isNew = false so it doesn't get flipped later. Getting that flag wrong caused mesh corruption during upsampling that was difficult to trace, so verifying it early saved significant debugging time.

Part 6: Loop subdivision for mesh upsampling

Implementation

Loop subdivision runs in five sequential passes. First, every existing vertex has its new position computed using the Loop weight formula, degree-3 vertices use u = 3/16 and all others use u = 3/(8n), blending the vertex's current position with its one-ring neighbor average. Second, every original edge stores the weighted midpoint position (3(a+b) + (c+d)) / 8 for the vertex that will be inserted there. Third, only the original edges are split (tracked via a pre-counted integer so the iterator doesn't walk into newly created edges), and each new midpoint vertex receives the position stored on its edge. Fourth, any new edge connecting exactly one new vertex to one old vertex is flipped to restore the correct triangulation. Finally, all newPosition values are committed.

The most important debugging insight was carefully managing the isNew flags. Specifically, the spoke edge that runs parallel to an original edge during a split must be marked isNew = false, otherwise the flip pass incorrectly processes it and corrupts the mesh. Using startIsNew != endIsNew as the flip condition was also cleaner and less error-prone than the equivalent double-negation boolean expression.

Behavior of Sharp Features After Subdivision

Loop subdivision smooths the mesh by pulling vertices toward their neighbors each iteration, so sharp corners and creases progressively round out. After just two or three iterations a cube's corners become visibly soft and its edges become curved. This is expected, Loop subdivision converges to a smooth limit surface, so any feature that isn't explicitly constrained will be averaged away.

Pre-splitting edges around a sharp feature before subdividing helps preserve it. By adding extra edge loops close to a corner or crease, the one-ring neighbors of vertices along that feature are pulled closer together, reducing how far the weighted average moves them. The effect is similar to adding holding edges in subdivision surface modeling.

Cube Asymmetry and How to Fix It

The default cube mesh has one diagonal edge per face, and all six diagonals happen to be oriented the same way relative to the cube's faces. This gives each face an asymmetric triangulation, one vertex on each face has degree 5 while the opposite has degree 3. Because the Loop weights depend on vertex degree, those two vertices move by different amounts each subdivision step, and the asymmetry compounds across iterations until the subdivided shape is noticeably lopsided.

The fix is to make every face's triangulation rotationally symmetric before subdividing. Adding a second diagonal to each face (so each face has an X pattern) gives every face vertex the same local connectivity. With uniform vertex degrees across the cube, every vertex moves by the same weighted amount and the shape remains symmetric under repeated subdivision. A single edge flip per face is sufficient on the default cube since each face already has one diagonal, flipping it to the opposite orientation and then adding the cross diagonal produces the symmetric starting configuration.
